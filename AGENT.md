# LLM RPG Server 开发代理规范

## 1. 文档目的

本文档用于约束和指导参与本仓库开发的 AI Agent。目标是在保持现有玩法可用的前提下，把当前平铺式原型逐步演进为边界清晰、低耦合、易测试、可替换基础设施的 RPG 服务。

本仓库当前包含角色、背包、交易、地图采集、PvE/PvP 战斗、LLM 裁判、炼金合成、NPC 对话、双向记忆、剧情钩子和世界事件等能力。现阶段以单体应用为主，不要求为了“分层”过早拆成微服务。

## 2. Agent 授权范围

Agent 被授权在当前项目目录内执行完成任务所需的代码与结构调整，包括：

- 新建、移动、重命名、拆分和删除项目文件；
- 重构模块、包、导入路径、配置和测试；
- 调整目录结构和应用启动方式；
- 增加依赖管理、迁移脚本、开发工具和文档；
- 在保持业务语义或明确说明变更的前提下，替换不合理的实现。

授权不等于可以忽略工程安全。Agent 必须：

- 开始前检查工作区状态，保留用户尚未提交的改动；
- 不使用 `git reset --hard`、强制覆盖或其他不可恢复的 Git 操作；
- 大范围移动前先识别入口、导入关系、测试和静态资源引用；
- 以小步、可验证的迁移方式工作，避免一次重写所有功能；
- 不修改项目目录之外的文件，除非用户另行明确授权；
- 涉及不可逆数据迁移、对外 API 破坏性变更、生产基础设施或需要产品决策的行为时，先向用户确认。

## 3. 核心架构原则

### 3.1 模块边界优先

代码按照业务领域组织，而不是按照技术类型把所有模型、服务或工具堆在同一层。推荐目标结构如下：

```text
llm-rpg-server/
├── pyproject.toml
├── README.md
├── AGENT.md
├── src/
│   └── llm_rpg_server/
│       ├── main.py                 # FastAPI 应用入口
│       ├── bootstrap.py            # 依赖装配与生命周期
│       ├── api/                    # HTTP/WebSocket 适配层
│       │   ├── dependencies.py
│       │   ├── error_handlers.py
│       │   └── routers/
│       ├── shared/                 # 少量跨域基础契约
│       │   ├── config/
│       │   ├── events/
│       │   └── observability/
│       ├── catalog/                # 职业、装备、道具、资源静态目录
│       ├── players/                # 玩家档案、背包与玩家仓储
│       ├── combat/                 # 战斗规则、状态图与房间协调
│       ├── crafting/               # 配方、扣料、生成与入库
│       ├── exploration/            # 世界、区域、地图、格子、采集
│       ├── npcs/                   # NPC、关系、记忆、对话、遭遇
│       └── world/                  # 世界事实、剧情事件及跨域编排
├── configs/
│   ├── catalog/
│   ├── prompts/
│   ├── narratives/
│   ├── npcs/
│   ├── maps/
│   └── schemas/
├── scripts/                        # 配置导入导出、校验和迁移
└── tests/
    ├── unit/
    ├── integration/
    ├── contract/
    └── fixtures/
```

该结构是目标而不是要求一次完成的模板。若实践中发现更合理的领域边界，可以调整，但必须保持依赖方向清晰。

### 3.2 依赖方向

每个领域内部优先采用以下层次：

```text
API / WebSocket adapter
          ↓
Application service / use case
          ↓
Domain model + deterministic rules
          ↓
Repository / provider interface
          ↓
Infrastructure implementation
```

约束：

- 领域模型不得依赖 FastAPI、LangChain、LangGraph、Langfuse 或具体数据库。
- API 层只负责协议校验、鉴权、调用用例和响应转换，不承载核心规则。
- 业务服务不得直接导入 `server.py`、应用入口或其他领域的全局字典。
- 跨领域协作通过明确的应用服务、仓储接口或领域事件完成。
- 禁止新增循环导入；现有 `craft -> server` 一类反向依赖应在迁移时优先移除。
- `shared` 只放真正稳定、被多个领域复用的最小契约，不得演变成杂物目录。

### 3.3 单体优先，接口可替换

短期保持模块化单体，避免不必要的网络边界。玩家、房间、配置、NPC 和 checkpoint 可以先有内存实现，但业务层必须依赖接口，使其可替换为数据库、Redis 或远程服务。

### 3.4 服务端权威与确定性规则

- 金币、背包、伤害、状态、奖励、触发资格等权威结果由服务端确定性代码决定。
- LLM 负责语言表达、叙事变体、受约束的内容生成，不直接决定资产和不可审计的核心数值。
- PvP 的胜负、命中和资源消耗不得依赖不可复现的自由文本判断。
- 重要流程应支持记录输入、随机种子、规则版本和输出，以便回放。

## 4. 静态文本与配置化规范

### 4.1 必须配置化的内容

以下内容不得继续以内联字符串散落在 Python 业务代码中：

- system prompt、user prompt 模板和结构化输出说明；
- NPC 人设、外貌、语气、背景、秘密、目标和初始关系；
- 剧情文本、世界观、故事钩子、遭遇对白和战报模板；
- 地图名称、区域描述、地标、环境和采集提示；
- 职业、武器、防具、物品、材料及其面向玩家的描述；
- 可运营调整的触发条件、掉落表、概率、数值和平衡参数；
- 面向玩家的错误提示、成功提示和默认降级台词。

代码中可以保留：

- 稳定的内部标识符、枚举值和协议字段名；
- 不面向玩家的开发日志；
- 无业务语义的技术异常信息；
- 为配置加载失败准备的最小安全错误，但不能用大段硬编码内容代替正式配置。

### 4.2 配置组织

配置以稳定 ID 关联，不以展示名称关联。推荐按用途拆分：

```text
configs/
├── catalog/       # characters, weapons, armors, items, resources
├── prompts/       # combat_judge, npc_dialogue, crafting
├── narratives/    # UI 文案、剧情和降级文本
├── npcs/          # NPC 档案、故事钩子和遭遇规则
└── maps/          # 世界、区域、地图模板、地形和生成参数
```

要求：

- 每类配置必须有 Pydantic 或 JSON Schema 校验。
- 配置加载失败应尽早失败并报告具体文件、字段和原因，不得静默吞掉错误。
- 路径以项目根目录或包资源为基准，不依赖进程启动目录。
- 配置包含 `schema_version`；需要兼容升级时提供迁移脚本。
- prompt 配置包含稳定 `prompt_id` 和版本信息，运行日志记录实际使用的版本。
- 密钥、令牌和服务地址属于运行环境配置，不写入静态内容文件。
- 中文为当前默认语言，但文案结构应为未来国际化保留 locale 层级或键值空间。

### 4.3 配置提供者接口

业务代码只依赖统一的配置接口，例如：

```python
class ContentProvider(Protocol):
    def get_prompt(self, prompt_id: str, version: str | None = None) -> PromptTemplate: ...
    def get_npc(self, npc_id: str) -> NPCDefinition: ...
    def get_map_template(self, template_id: str) -> MapTemplate: ...
    def get_text(self, text_id: str, locale: str = "zh-CN") -> str: ...
```

实现顺序：

1. `LocalContentProvider`：当前默认，从仓库内静态文件加载。
2. `CachedContentProvider`：为配置增加进程内或 Redis 缓存。
3. `RemoteContentProvider`：未来从 Langfuse 或运营后台 API 获取。
4. 远程配置不可用时，可按明确策略回退到已验证的本地版本或最后一次有效缓存。

不得在各业务模块中直接散落 Langfuse/API 获取逻辑。远程内容来源只能由 provider/infrastructure 层接入。

### 4.4 Prompt 安全边界

- Prompt 中明确模型权限，禁止模型直接发放奖励、修改库存或决定未授权剧情完成状态。
- 传入模型的 NPC 信息按公开/私密权限裁剪。
- 模型输出必须经过结构化 schema 校验及确定性业务校验。
- 日志不得记录密钥；涉及玩家输入和 NPC 私密设定时应考虑脱敏及可观测性范围。

## 5. 地图系统升级规范

### 5.1 地图层级与规模

地图不再固定为单一 5×5 数组，应至少支持下列层级：

```text
World（世界）
└── Region（区域，例如王国、荒漠、群岛）
    └── MapInstance（地图实例）
        └── Cell / Location（格子或地点）
```

地图模板增加规模类别。默认建议：

- `small`：适合单次探索或室内区域；
- `medium`：常规野外区域；
- `large`：长期探索区域；
- `custom`：显式指定宽、高或非网格拓扑。

规模类别对应的默认尺寸必须由配置决定，代码不得写死。数据模型至少应表达：

- `world_id`、`region_id`、`map_id`、`template_id`；
- `scale`、`width`、`height` 或拓扑类型；
- 地形、地标、可通行性、已探索/已采集状态；
- 生成规则、随机种子和配置版本；
- 玩家当前地图位置和地图实例生命周期。

地图生成规则与玩家探索进度分离：模板是静态定义，实例是运行时状态。

### 5.2 NPC 与区域关联

NPC 不应只通过一个 `terrain_id` 和若干 `cell_id` 静态出现。引入可配置的 `NPCSpawnRule` 或 `EncounterRule`，至少支持：

- NPC 可出现的世界、区域、地图模板、地形、地标或坐标范围；
- 出现概率和权重；
- 固定出现、随机偶遇、进入区域触发、移动到格子触发等触发方式；
- 时间段、天气、玩家等级、任务状态、关系值、记忆标签和世界事实条件；
- 冷却时间、触发次数、是否唯一、是否可重复；
- 触发后的剧情钩子、对话入口、战斗入口或世界事件；
- 冲突处理和优先级。

示意配置：

```yaml
encounter_id: encounter.darok.desert_road
npc_id: darok_blacksalt
locations:
  region_ids: [western_desert]
  terrain_ids: [desert]
  landmark_ids: [salt_road]
trigger: on_enter_cell
chance: 0.2
priority: 50
conditions:
  - kind: relationship_at_most
    field: affinity
    threshold: 20
cooldown:
  scope: player
  seconds: 86400
repeatable: true
story_hook_id: hook.darok.lost_manifest
```

### 5.3 遭遇解析流程

推荐由独立 `EncounterService` 解析：

```text
玩家进入区域/格子
       ↓
探索服务产生 LocationEntered 事件
       ↓
EncounterService 获取候选规则
       ↓
确定性条件过滤 + 带种子的概率选择
       ↓
生成 Encounter
       ↓
对话 / 剧情钩子 / 战斗 / 世界事实
```

地图模块只描述空间和移动，不直接调用 LLM；NPC 模块只管理 NPC 领域状态，不负责地图生成。跨域流程由应用服务或领域事件协调。

### 5.4 兼容迁移

现有玩家的 5×5 `current_map` 数据应通过适配器或迁移函数转换为新的 `MapInstance`，不要在同一业务代码中长期并存两套结构。迁移期可保留兼容读取，但必须标注移除条件和测试覆盖。

## 6. 数据与一致性

- 玩家、库存、配方、地图实例、NPC 关系、记忆和世界事实均通过仓储访问。
- 交易、合成扣料、奖励结算等跨多字段写操作必须具备原子性。
- 库存使用统一条目模型；不得同时用整数和对象表示同一类材料数量。
- 配方主键至少包含物品类型和物品 ID，避免不同目录间 ID 碰撞。
- WebSocket 连接必须验证玩家身份及房间成员资格。
- 配装、使用道具、售卖、合成时必须在服务端验证所有权和数量。
- 房间动作提交、结算和奖励应具备幂等键或版本检查，避免并发重复执行。
- 公共 API 不得暴露 NPC 私密背景、内部 prompt 或仅供规则层使用的字段。

## 7. API 与兼容性

- 新接口使用明确的 Pydantic 请求/响应模型，不使用裸 `dict` 作为业务协议。
- 枚举字段使用 `Enum` 或 `Literal`，并提供稳定的机器可读错误码。
- 破坏现有接口前，先评估调用方；必要时提供版本化路由或兼容期。
- HTTP、WebSocket 和内部领域事件应共享明确的事件名称与 schema，但不得共享可变对象。
- API 层统一处理领域异常到 HTTP/WebSocket 错误的映射。

## 8. 测试与质量门槛

每次结构性重构必须补充或保持以下验证：

- 配置 schema、重复 ID、悬空引用和 prompt 变量检查；
- 玩家、库存、交易、合成和结算单元测试；
- 地图规模、确定性生成、探索进度和采集测试；
- NPC 条件过滤、偶遇概率、冷却、唯一性和玩家隔离测试；
- 战斗状态图路由和幂等结算测试；
- FastAPI 路由与 WebSocket 身份/房间权限集成测试；
- 本地内容 provider 与远程 provider 的契约测试；
- 模型或远程内容服务不可用时的降级测试。

测试应避免真实调用 LLM、图片服务和 Langfuse；使用受控 fake/stub。至少为关键规则提供确定性随机种子。

在交付改动前，Agent 应运行仓库已有的格式化、静态检查和测试命令。若因依赖或环境缺失无法运行，必须明确报告未验证范围，不能声称测试通过。

## 9. 推荐迁移顺序

为避免大爆炸式重写，按以下顺序推进：

1. 建立依赖清单、`src` 包、应用入口和测试基线。
2. 抽出玩家仓储、目录配置和统一库存模型，消除业务模块对 `server.py` 的反向导入。
3. 拆分 `server.py` 为领域 router、应用服务和 WebSocket 房间协调器。
4. 引入 `ContentProvider`，迁移 prompt、NPC、剧情、地图和 UI 文案。
5. 迁移战斗与合成图，使节点依赖显式注入的服务，而非全局对象。
6. 实现分层地图模型、规模模板和地图实例。
7. 实现 NPC encounter 配置、条件解析和位置事件。
8. 接入数据库/Redis，并增加事务、幂等和多实例支持。
9. 最后再评估 Langfuse 或后台内容 API；本地 provider 始终保留为开发和故障回退能力。

每一步都应保持应用可启动、接口行为可验证，并尽量把纯移动与行为变化分开提交。

## 10. Agent 工作方式

接到开发任务时，Agent 应：

1. 阅读本文件、相关模块、配置和测试。
2. 检查 Git 状态，识别用户已有改动。
3. 说明本次改动所属领域、边界和兼容策略。
4. 优先复用既有领域模型和接口，不重复创建相似抽象。
5. 实施最小完整变更，并同步更新配置、schema、测试和文档。
6. 运行相应验证，报告结果和仍存在的风险。

遇到以下不确定性必须先向用户确认：

- 地图规模的具体尺寸、是否支持六边形或图结构等产品规则；
- NPC 偶遇是否要求严格可复现，以及概率、冷却和唯一性的产品定义；
- Langfuse 与未来后台谁是内容真源，离线时采用何种回退策略；
- 是否允许破坏现有 API、玩家存档或配置格式；
- 需要引入哪种数据库、缓存、任务队列或身份认证方案；
- 变更会删除用户数据、迁移生产数据或影响外部系统。

对于可以通过仓库现状判断、风险较低且不改变产品语义的问题，Agent 应做出合理假设继续推进，并在交付说明中列出假设，不必为每个实现细节阻塞开发。

## 11. 完成定义

一项改动只有在满足以下条件时才算完成：

- 代码位于合理的领域和层次中，没有新增循环依赖或隐式全局耦合；
- 新增面向玩家的文本和 prompt 已配置化；
- 配置有 schema、稳定 ID 和引用校验；
- 关键业务规则不由 LLM 自由决定；
- 输入、权限、资产所有权和并发路径经过校验；
- 相关测试已补充并通过，或清楚说明无法执行的原因；
- 文档、启动方式和迁移说明与实现一致；
- 未破坏用户未提交的改动，也未留下无说明的兼容层或临时代码。
